# IslandCore 开发与商业化全纲领

**技术栈（全阶段统一）：我们采用 Rust + Tauri。**  
桌面壳与系统集成由 Rust 负责；灵动岛 UI、动画与主题由 Tauri WebView 内前端（如 Svelte/React/Vue + CSS）实现，必要时通过 Tauri 命令与 Rust 侧通信。

---

## 第一阶段：基座搭建 (The Foundation)

**目标：** 在 Windows 上实现一个能「呼吸」的透明悬浮窗。

**我们采用 Rust + Tauri：**

- **环境配置：** 安装 Rust 工具链、Node（或 Bun）用于前端构建；Windows 上安装 Visual Studio Build Tools（C++ 桌面开发）以满足 Tauri 与 `windows` crate 的链接需求。
- **项目创建：** 使用 `create-tauri-app` 创建 Tauri 2 项目；窗口行为在 `tauri.conf.json` / Rust `Window` API 中配置。
- **窗口隐身术：** 在 Rust 侧通过 `windows` / `raw-window-handle` 取得 HWND，调用 `SetWindowLong` / `SetWindowLongPtr` 等移除标题栏、边框、任务栏图标（或结合 Tauri 无边框、透明选项）。
- **置顶与透明度：** 使用 `WS_EX_LAYERED`、`WS_EX_TRANSPARENT`（按需）与 `SetLayeredWindowAttributes` 等实现置顶与 ExStyle 透明度；与 Tauri 的 `transparent`、始终置顶 API 对齐。
- **核心画布层：** WebView 内用 HTML/CSS（Flex/Grid）定义胶囊形灵动岛基础形状；高刷下动画以 `requestAnimationFrame` + CSS `transform`/`opacity` 为主，避免布局抖动。
- **材质感：** 用 CSS `backdrop-filter`、渐变与半透明层模拟 Mica/Acrylic 观感；若需与系统 Mica 完全一致，可再评估 `window-vibrancy` 类插件或 HWND 上的 DWM 扩展。

---

## 第二阶段：灵魂动画 (The Motion Engine)

**目标：** 实现「果冻感」和「丝滑感」，这是卖 $3.99 的核心价值。

**我们采用 Rust + Tauri：**

- **动画主战场：** 以 Web 技术栈实现为主（CSS `@keyframes`、Web Animations API、或轻量动画库），保证与 UI 同层、易迭代。
- **物理感：** 用弹簧曲线参数（等价于阻尼 DampingRatio、刚度 Stiffness）在 JS/TS 或 WASM 中实现缓动；复杂物理可引入小型弹簧求解或贝塞尔近似。
- **流体形变：** 胶囊 ↔ 矩形通过 `border-radius`、`width`/`height`、`transform: scale()` 的非线性过渡；避免每帧改 `box-shadow` 等重属性。
- **内容联动：** 文字/图标随形变同步 `opacity`、`translate`；大段内容用 `IntersectionObserver` 或简单状态机控制显隐。
- **交互反馈：** Hover 轻微缩放与回弹；Click 展开用分段动画或 FLIP 技术；Rust 侧仅负责触发事件与性能敏感时的节流。

## 第三阶段：感知系统 (The Synchronizer)

**目标：** 让灵动岛「读懂」整个 Windows 系统。

**我们采用 Rust + Tauri：**

- **媒体同步：** Rust 调用 Windows Runtime / Win32（如 `GlobalSystemMediaTransportControls` 相关 API，或通过 `windows` crate 封装）获取播放状态与封面；封面图解码后提取主色调（palette 算法在 Rust 或前端完成），经 Tauri 命令/events 推给 WebView 做边缘微光。
- **通知接管：** 使用 `UserNotificationListener` 等 UWP API（Rust 中通过 `windows` crate 或单独辅助进程）监听通知；与「通知吸入」动画联动：原生 toast 消失时机与前端膨胀动画由事件序列对齐。
- **系统监控：** 音量、静音、电池、CPU 等通过 Rust 调用对应 Win32/WMI/性能计数器；结果以轮询或事件形式注入前端，控制灵动岛上的实时展示与充电动画等。

## 第四阶段：商业化增强 (Steam Ready)

**目标：** 达到上架 Steam 的工业强度。

**我们采用 Rust + Tauri：**

- **多语言与国际化：** 前端用 `i18next` / `fluent` 等方案；Rust 侧错误与系统字符串可用 `fluent`/`gettext` 或资源文件；覆盖中、英、日、法等主流市场。
- **Steamworks SDK 集成：** 在 Rust 中绑定或封装 Steamworks C API（如 `steamworks-rs` 等成熟封装），处理成就、创意工坊下载路径与用户内容目录。
- **创意工坊：** 约定 `Theme.json`（或等价格式）皮肤接口；Rust 校验与沙箱读取路径，前端按主题包渲染。
- **成就感：** 设计如「多任务大师」（同时处理 3 个任务流）等成就，与 Steam 成就 ID 映射并在 Rust 侧解锁。
- **性能压测：** Rust 侧控制常驻内存与后台轮询频率；WebView 侧避免泄漏（及时取消监听、复用 Canvas）；目标常驻内存可控（例如努力控制在约 60MB 量级，以实测为准）。

fn sense_world() -> CurrentContext {
    let active_app = get_active_window_info();
    
    // 1. 检查是否有“抢占式”通知（如验证码、低电量）
    if let Some(urgent) = get_urgent_event() {
        return Context::Urgent(urgent);
    }

    // 2. 检查是否有“持续性”背景任务（如音乐、下载）
    let bg_tasks = get_background_tasks();
    
    // 3. 综合判断形态
    match (active_app, bg_tasks) {
        (App::Chrome, tasks) if tasks.is_empty() => Context::WebMode,
        (App::Game, _) => Context::GamingMode,
        (_, tasks) if !tasks.is_empty() => Context::MediaFocus,
        _ => Context::Idle,
    }
}
